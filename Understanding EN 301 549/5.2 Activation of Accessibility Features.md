# Understanding 5.2 Activation of Accessibility Features

Status of this document: 
⚠️ Draft mode.


Normative requirement (EN 301 549 v4.1.0 draft)
> "Accessibility features shall be discoverable and operable without requiring assistance from another person or requiring interaction methods that are not accessible to users with disabilities."


## Intent
The intent of this requirement is to ensure that accessibility features are **practically usable in real-world condition**s, not just technically present.

Many products include accessibility-related functionality (such as screen reader support, text resizing, captions, or contrast adjustments), but fail when users cannot easily locate or enable these features.

This requirement addresses barriers such as:

- Accessibility settings hidden deep within complex navigation structures.
- Features that require sighted assistance to activate.
- Reliance on inaccessible controls (e.g., drag-only gestures without keyboard alternatives).
- Lack of discoverability for users who need accessibility features at the point of first use or urgent need.

**This ensures users can activate accessibility features independently at the point of need.**

## Requirement
Where a product or service provides accessibility features or settings, those features shall be:

- discoverable
- operable
- and accessible via assistive technologies
**without requiring assistance from another person or requiring interaction methods that are not accessible to users with disabilities.**

Activation of each accessibility feature shall be possible using **at least one interaction method that is not limited to a single sensory channel or a single input modality**.


## This requirement benefits
This requirement benefits a broad range of users, including:

### Sensory disabilities
- Blind and low vision users relying on screen readers or magnification.
- Deaf and hard of hearing users requiring captions or visual alerts.

### Motor disabilities
- Users with motor impairments who rely on keyboard navigation, switches, or voice control.

### Cognitive and neurological disabilities
- Users requiring simplified interfaces or reduced motion settings.

### Situational or temporary impairments
- Injury (temporary loss of mobility or vision)
- Environmental conditions (bright sunlight, noisy environments, one-handed use)

### Older users
- Users with age-related changes in vision, hearing, or motor ability.


## Examples

### Good examples
- A clearly labeled **“Accessibility” section in system settings that is directly accessible from primary navigation or system settings**.
- A setup wizard that prompts users to enable high contrast or screen reader support during first use.
- Keyboard-accessible toggle for captions available in media players.
- Voice assistant command that directly enables a screen reader **without requiring visual confirmation steps**.
- Persistent accessibility shortcut (e.g., triple-tap or shortcut key) that is documented and usable without vision.

### Problematic examples
- Accessibility features requiring access to developer menus, making them undiscoverable during normal product use
- Screen reader support requiring inaccessible touch-only gestures without equivalent alternatives
- High contrast mode hidden in unrelated visual settings without clear semantic labeling
- Features requiring login to a separate website or support request to enable
- Accessibility settings only discoverable through visual-only onboarding flows

## Evaluation methods
Evaluation determines whether accessibility features can be located, understood, and activated independently by users with disabilities, including users relying on assistive technologies.

Assessment includes:

- Discoverability of accessibility settings and features.
- Accessibility of activation controls themselves.
- Independence of activation (no external assistance required).
- Consistency across platforms (web, mobile, desktop).
- Availability of equivalent activation methods across input modalities.
- Ability to activate accessibility features using assistive technology alone.

### Failures
A product fails this requirement when:

- Accessibility features cannot be reached using assistive technologies (e.g., screen reader focus cannot access settings).
- Activation requires sighted assistance or external support.
- Features are hidden behind non-accessible onboarding or tutorial flows.
- Activation requires a single interaction method without equivalent alternatives.
- Accessibility settings are not clearly identified as accessibility-related or are not semantically labeled to their function.
- Critical accessibility features require third-party installation or external configuration.

### Passing techniques
A product meets this requirement when it:

- An accessibility settings area shall be clearly labeled and directly accessible from primary navigation or system settings.
- All accessibility features shall be reachable via keyboard and assistive technologies.
- Multiple activation methods shall be available (e.g., settings menu, shortcuts, voice control).
- Accessibility shortcuts shall be documented and consistently available across the product.
- Onboarding flows shall not block or delay activation of accessibility features.
- Accessibility functions shall use clear and consistent terminology across the interface.

### Testing methods
Testing may include:

- Navigating the interface using only a screen reader.
- Navigating using keyboard only (no mouse or touch input)
- Verifying accessibility feature discoverability from first interaction.
- Checking availability of alternative activation methods (keyboard, voice, gesture).
- Reviewing onboarding and setup flows for accessibility barriers.
- Confirming consistency across devices and platforms.
- Attempting activation without prior knowledge of system structure.
- Verifying that activation does not require sighted assistance.
- Verifying equivalent activation across input modalities.

### Key Terms
**Accessibility features:** Functionalities that support users with disabilities, such as screen readers, captions, zoom, or input alternatives

**Activation:** The process of enabling or turning on a feature.

**Discoverability:** The ease with which users can find a feature without prior knowledge.

**Assistive technology:** External tools such as screen readers, switch devices, or voice control systems

**Independent use:** The ability to complete a task without assistance from another person

**Input modality:** The method used to interact with a system (e.g., keyboard, touch, voice)
