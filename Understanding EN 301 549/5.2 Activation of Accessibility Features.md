# Understanding 5.2 Activation of Accessibility Features

Status of this document: ⚠️ Draft mode.

> Normative requirement (EN 301 549 v4.1.0 draft)


## Intent
The intent of this requirement is to ensure that accessibility features are **practically usable in real-world** conditions, not just technically present.

Many products include accessibility-related functionality (such as screen reader support, text resizing, captions, or contrast adjustments), but fail when users cannot easily locate or enable these features.

This requirement addresses barriers such as:

- Accessibility settings hidden deep within complex navigation structures
- Features that require sighted assistance to activate
- Reliance on inaccessible controls (e.g., drag-only gestures without keyboard alternatives)
- Lack of discoverability for users who need accessibility features most urgently (including first-time setup scenarios)

The requirement ensures that accessibility features are **not gated behind usability barriers**, especially at the point where users most need them.

## Requirement

Where a product or service provides accessibility features or settings, those features **shall be identifiable, discoverable, and operable by users, including users relying on assistive technologies, without requiring unreasonable effort or third-party assistance**.

Activation of accessibility features shall be possible through **at least one method that does not depend solely on a specific sensory modality or a single interaction path, and shall not require inaccessible or hidden activation steps**.


## This requirement benefits
This requirement benefits a broad range of users, including:

- Blind and low vision users relying on screen readers or magnification
- Deaf and hard of hearing users needing captions or visual alerts
- Users with motor impairments who rely on keyboard navigation, switches, or voice control
- Users with cognitive disabilities who may require simplified interfaces or reduced motion settings
- Temporary disability situations (e.g., injury, situational impairments)
- Older users who may have reduced sensory or motor capabilities
- Users in constrained environments (bright sunlight, noisy environments, one-handed use)

## Examples
Good examples:

- A clearly labeled “Accessibility” section in system settings with all accessibility features grouped logically
- A setup wizard that prompts users to enable high contrast or screen reader support during first use
- Keyboard-accessible toggle for captions available in media players
- Voice assistant command: “Turn on screen reader”
- Persistent accessibility shortcut (e.g., triple-tap or shortcut key) that is documented and usable without vision

Problematic examples:

- Accessibility features only accessible through developer menus
- Screen reader support that must be enabled via inaccessible touch gestures without alternatives
- High contrast mode hidden in unrelated visual settings without labeling
- Features requiring login to a separate website or support request to enable
- Settings that are only discoverable through visual-only onboarding steps

## Evaluation methods
Evaluation involves determining whether accessibility features can be **located, understood, and activated independently by users with disabilities**, including those using assistive technologies.

Assessment should consider:

- Discoverability of accessibility settings
- Accessibility of activation controls themselves
- Independence of activation (no external assistance required)
- Consistency across platforms (web, mobile, desktop)

### Failures
A product fails this requirement when:

- Accessibility features exist but are not reachable via assistive technology
- Activation requires sighted assistance or external support
- Features are hidden behind non-accessible onboarding or tutorials
- Only one interaction method exists (e.g., gesture-only activation without keyboard equivalent)
- Accessibility settings are not labeled or are ambiguous in meaning
- Critical accessibility features require third-party installation or configuration

### Passing techniques
A product meets this requirement when it:

- Provides a clearly labeled accessibility settings area
- Ensures all accessibility features are reachable via keyboard and assistive technologies
- Offers multiple activation paths (e.g., settings menu, shortcuts, voice control)
- Includes accessibility shortcuts that are documented and consistently available
- Ensures onboarding processes do not block or delay activation of essential accessibility features
- Uses clear, consistent terminology for accessibility functions

### Testing methods
Testing may include:

- Navigating the interface using only a screen reader
- Navigating using keyboard only (no mouse or touch)
- Verifying accessibility feature discoverability from first user interaction
- Checking shortcut and alternative activation methods
- Reviewing onboarding and setup flows for accessibility barriers
- Confirming consistency of access across devices and platforms
- Attempting activation without prior familiarity with the system

### Key Terms
- Accessibility features: Functionalities that support users with disabilities, such as screen readers, captions, zoom, or input alternatives
- Activation: The process of enabling or turning on a feature
- Discoverability: The ease with which users can find a feature without prior knowledge
- Assistive technology: External tools such as screen readers, switch devices, or voice control systems
- Independent use: The ability for a user to complete a task without assistance from another person
- Non-sensory dependent interaction: Interaction not limited to a single sensory channel (e.g., not only visual or only touch-based)
